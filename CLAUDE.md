## Project Overview

FDP is a full-stack food delivery platform (similar to UberEats/DoorDash). It is a monorepo with:
- **Backend**: Node.js + Express + MongoDB (Mongoose ODM)
- **Frontend**: Vanilla JavaScript + HTML/CSS (no framework, no build step)

## Development Commands

### Backend
```bash
cd backend
npm install
npm run seed      # Seed MongoDB with restaurants/foods from frontend/data/
npm run dev       # Start dev server with nodemon (http://localhost:5000/api)
npm start         # Production start
```

### Frontend
Serve `frontend/` as static files on port 5500:
```bash
cd frontend
python -m http.server 5500
# or: npx serve -l 5500 .
```

### Environment Setup
Copy `backend/.env.example` to `backend/.env`. Required variables:
```
PORT=5000
MONGODB_URI=mongodb://127.0.0.1:27017/fdp
JWT_ACCESS_SECRET=...
JWT_REFRESH_SECRET=...
FRONTEND_ORIGIN=http://localhost:5500
```

No test suite or linter is configured.

## Architecture

### Backend Structure

```
backend/src/
  server.js        — entry point, starts HTTP server
  app.js           — Express app: CORS, middleware, mounts all routes
  config.js        — reads .env into exported config object
  database.js      — Mongoose connection
  data/store.js    — business logic layer (all DB queries, auth helpers)
  middleware/auth.js— JWT verification; exports requireAuth and requireAdmin
  models/          — 10 Mongoose schemas (User, Food, Restaurant, Order, Cart, Review, DeliveryPerson, Promotion, RefreshToken, Counter)
  routes/
    auth.js        — /api/auth/* (register, login, refresh, logout, profile)
    restaurants.js — /api/restaurants/* and /api/foods/* (browse, search, reviews)
    orders.js      — /api/orders/* and /api/cart/* (cart, create/list orders)
    admin.js       — /api/admin/* (order management, delivery, promotions)
  utils/
    counters.js    — auto-incrementing integer IDs for MongoDB documents
    formatters.js  — serializes Mongoose docs to plain API response objects
  scripts/seed.js  — seeds DB from frontend/data/restaurants.json and foods.json
```

**`store.js` is the central data layer.** All routes call functions exported from `store.js` rather than querying models directly. When adding features, add the DB logic here.

**`counters.js` manages auto-increment IDs.** All models use integer IDs (`user_id`, `food_id`, etc.) generated by `getNextId(key)` rather than MongoDB ObjectIds as the primary business key.

**`formatters.js` controls API shape.** Serializer functions live here—never send raw Mongoose documents to the client.

### Authentication

- Access token: 1-day JWT, sent in `Authorization: Bearer` header
- Refresh token: 7-day JWT, stored in MongoDB `RefreshToken` collection
- `requireAuth` middleware verifies access token and attaches `req.user`
- `requireAdmin` additionally checks `req.user.role === 'admin'`

### Frontend Structure

```
frontend/
  pages/           — 8 HTML files (one per page)
  assets/
    js/
      api.js       — API client: all fetch calls to the backend
      main.js      — core page logic (restaurants, menu, browsing, cart UI)
      auth.js      — login/register form handling
      cart.js      — cart management & checkout
      admin.js     — admin dashboard
      settings.js  — user profile settings
    css/           — per-page CSS files
    images/foods/  — food item images
  data/
    restaurants.json — source data for seeding
    foods.json       — 80+ food items used by seed script
```

The frontend stores the access token, refresh token, and user object in `localStorage`. `api.js` handles token attachment and refresh logic for all API calls.

### Data Seeding

`npm run seed` reads `frontend/data/restaurants.json` and `frontend/data/foods.json`, then writes them into MongoDB. It also creates a default admin user. Run this once after a fresh database.

## Key Design Decisions

- **Integer IDs over ObjectIds**: All resources use auto-incremented integer IDs (e.g., `order_id: 42`) for cleaner API responses. The `Counter` model tracks sequence values per resource type.
- **Cart in localStorage**: Cart is currently stored client-side and only synced to the backend at checkout. Server-side cart sync is planned for a future phase.
- **Single restaurant**: The seed data loads one restaurant ("Pizzaburg") with multiple food categories. The multi-restaurant UI is already implemented and ready for more seed data.

## Role Hierarchy (added 2026-04-11)

Four roles exist, lowest to highest:
- `user` — regular customer
- `admin` — existing dashboard (`admin.html`), manages orders/delivery/promotions
- `restaurant_admin` — manages their own restaurant via `partner-dashboard.html`
- `superadmin` — highest tier, manages applications + restaurant listings via `superadmin.html`

Default credentials after `npm run seed`:
- Admin: `admin@fdp.com` / `admin123`
- Superadmin: `superadmin@fdp.com` / `superadmin123`

## Partner Onboarding Flow (added 2026-04-11)

1. Business submits application at `partner-apply.html`
2. Superadmin approves in `superadmin.html` → system creates a `restaurant_admin` user + Restaurant doc
3. Temp password is returned **once** in the API response — superadmin must share it with the partner
4. Partner logs in → redirected to `partner-dashboard.html` to manage their restaurant

New files added:
- `backend/src/models/PartnerApplication.js`
- `backend/src/routes/partner.js` — `/api/partner/*`
- `backend/src/routes/restaurant-admin.js` — `/api/restaurant/*`
- `frontend/pages/partner-apply.html` + `partner-dashboard.html` + `superadmin.html`
- `frontend/assets/js/partner-dashboard.js` + `superadmin.js`
- `frontend/assets/css/partner-apply.css` + `partner-dashboard.css` + `superadmin.css`

## Superadmin Features Roadmap

Features already built (✅) and remaining (📋):

### ✅ Done
- Partner application approval/rejection (with rejection reason)
- Restaurant listing management (restrict/unrestrict, delete)
- User management — search, ban/unban (with reason), reset password

### 📋 Next: Platform Analytics
Show platform-wide stats on the superadmin dashboard:
- Total orders and revenue across all restaurants
- User growth over time
- Most popular restaurants
- No per-restaurant breakdown (that's restaurant_admin territory)

### 📋 Next: Audit Log
Track every superadmin action with a lightweight `AuditLog` model:
- Fields: `actor_id`, `action` (e.g. `ban_user`, `approve_application`), `target_id`, `target_type`, `note`, `created_at`
- Show in a dedicated "Audit Log" tab in `superadmin.html`
- Essential for accountability and dispute resolution

### 📋 Next: Global Promotions
Platform-wide discount codes managed by superadmin (distinct from per-restaurant promos managed by admin):
- Superadmin creates codes with flat or percentage discounts, expiry dates, usage limits
- Applied at checkout before restaurant-level promos

### 📋 Next: Category Management
Centrally define food categories that restaurant admins select from when adding items:
- Superadmin CRUD for categories
- `Food` model references category by ID instead of free-text string

### 📋 Next: Announcement / Broadcast
Send a site-wide notice to all users (maintenance windows, new features):
- Stored in DB, shown as a dismissible banner on the frontend
- Superadmin creates/deactivates announcements from dashboard

### 📋 Next: Commission / Billing Config
Set platform fee percentage per restaurant:
- Stored on the Restaurant doc (`commission_rate`)
- Groundwork for monetization; displayed in restaurant_admin dashboard

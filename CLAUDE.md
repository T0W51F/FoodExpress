## Project Overview

FDP is a full-stack food delivery platform (similar to UberEats/DoorDash). It is a monorepo with:
- **Backend**: Node.js + Express + MongoDB (Mongoose ODM)
- **Frontend**: Vanilla JavaScript + HTML/CSS (no framework, no build step)

## Development Commands

### Backend
```bash
cd backend
npm install
npm run seed      # Seed MongoDB with restaurants/foods from frontend/data/
npm run dev       # Start dev server with nodemon (http://localhost:5000/api)
npm start         # Production start
```

## Architecture

### Backend Structure

```
backend/src/
  server.js              — entry point, starts HTTP server
  app.js                 — Express app: CORS, middleware, mounts all routes
  config.js              — reads .env into exported config object
  database.js            — Mongoose connection
  data/store.js          — business logic layer (all DB queries, auth helpers)
  middleware/auth.js     — JWT verification; exports requireAuth, requireAdmin, requireRestaurantAdmin
  models/                — Mongoose schemas (User, Food, Restaurant, Order, Cart, Review,
                           DeliveryPerson, Promotion, RefreshToken, Counter, PartnerApplication)
  routes/
    auth.js              — /api/auth/* (register, login, refresh, logout, profile)
    restaurants.js       — /api/restaurants/* and /api/foods/* (browse, search, reviews)
    orders.js            — /api/orders/* and /api/cart/* (cart, create/list orders)
    admin.js             — /api/admin/* (superadmin-level management; keep — used by superadmin.html)
    partner.js           — /api/partner/* (applications)
    restaurant-admin.js  — /api/restaurant/* (per-restaurant management, requireRestaurantAdmin)
  utils/
    counters.js          — auto-incrementing integer IDs for MongoDB documents
    formatters.js        — serializes Mongoose docs to plain API response objects
  scripts/seed.js        — seeds DB from frontend/data/restaurants.json and foods.json
```

**`store.js` is the central data layer.** All routes call functions exported from `store.js` rather than querying models directly. When adding features, add the DB logic here.

**`counters.js` manages auto-increment IDs.** All models use integer IDs (`user_id`, `food_id`, etc.) generated by `getNextId(key)` rather than MongoDB ObjectIds as the primary business key.

**`formatters.js` controls API shape.** Serializer functions live here — never send raw Mongoose documents to the client.

### Authentication

- Access token: 1-day JWT, sent in `Authorization: Bearer` header
- Refresh token: 7-day JWT, stored in MongoDB `RefreshToken` collection
- `requireAuth` verifies access token and attaches `req.user`
- `requireAdmin` checks `req.user.role === 'admin'`
- `requireRestaurantAdmin` checks role is `restaurant_admin` or `superadmin`

### Frontend Structure

```
frontend/
  pages/           — 11 HTML files
    index.html            — homepage / restaurant listing + search
    menu.html             — restaurant menu page
    restaurants.html      — browse all restaurants
    cart.html             — cart & checkout
    orders.html           — order history
    login.html / register.html / settings.html
    partner-apply.html    — partner application form
    partner-dashboard.html — restaurant admin dashboard (9 tabs)
    superadmin.html       — superadmin dashboard
  assets/
    js/
      api.js              — API client: fetch with Bearer auth + token refresh
      main.js             — homepage/browse logic, nav, user menu
      auth.js             — login/register form handling
      cart.js             — cart management & checkout
      partner-dashboard.js — restaurant admin dashboard logic
      superadmin.js       — superadmin dashboard logic
      settings.js         — user profile settings
    css/                  — per-page CSS files
    images/foods/         — food item images
  data/
    restaurants.json      — source data for seeding
    foods.json            — 80+ food items used by seed script
```

The frontend stores access token, refresh token, and user object in `localStorage`. `api.js` handles token attachment and refresh logic for all API calls.

### Data Seeding

`npm run seed` reads `frontend/data/restaurants.json` and `frontend/data/foods.json`, then writes them into MongoDB. Also creates default superadmin and restaurant admin accounts. Run once after a fresh database.

## Key Design Decisions

- **Integer IDs over ObjectIds**: All resources use auto-incremented integer IDs (e.g., `order_id: 42`) for cleaner API responses. The `Counter` model tracks sequence values per resource type.
- **Cart in localStorage**: Cart is stored client-side and synced to the backend at checkout only.
- **Shared delivery pool**: All restaurant admins manage the same driver list — per-restaurant pools are out of scope.
- **Promotions**: Currently global (no restaurant_id scoping on Promotion model) — all restaurant admins share the same promo pool.

## Search (index.html + restaurants.html)

All client-side. No backend search endpoint.

**Hero autocomplete** (`initHeroSearch` in `main.js`):
- Fetches all restaurants + foods once on first keystroke, cached in `window.searchCache`
- Shows dropdown with 3 sections: Restaurants (→ menu page), Categories (→ filtered restaurants), Foods (→ search results)
- Fuzzy matching via `fuzzyMatch()` — subsequence check
- Dropdown appended to `document.body` (not the form) — escapes `overflow:hidden` + `backdrop-filter` containing block on hero
- `position: fixed`, repositioned on scroll/resize via `positionDropdown()`
- Keyboard nav: ↑↓ navigate, Esc close, Enter select

**Ranked search** (`filterRestaurants` in `main.js`):
- `scoreRestaurantMatch()` returns numeric relevance score per restaurant
- Score tiers: exact name (100) → name contains (80) → cuisine/category (60/40) → food items (40/30/10) → fuzzy (20)
- Relevance order preserved when sort dropdown is on default "rating"; explicit sort overrides it
- `?cuisine=X` URL param pre-selects the cuisine filter on page load

## Role Hierarchy

Four roles, lowest to highest:
- `user` — regular customer
- `restaurant_admin` — manages their own restaurant via `partner-dashboard.html`
- `superadmin` — platform management via `superadmin.html`

> The `admin` role is retired. `admin.html`, `admin.js`, `admin.css` have been deleted.
> Do not recreate them or add routes requiring `role === 'admin'`.

## Partner Dashboard (partner-dashboard.html)

Nine-tab sidebar dashboard for `restaurant_admin` role. Tabs:

| Tab | Type | Description |
|-----|------|-------------|
| Overview | read | Stat cards (orders, revenue, active orders, menu items), recent orders feed, popular items |
| Menu | CRUD | Food card grid, add/edit/delete, add-ons editor |
| Profile | edit | Restaurant info, operating hours, open/closed toggle |
| Orders | action | Table with inline status dropdown + Cancel button; PATCH /restaurant/orders/:id/status |
| Deliveries | CRUD | Shared driver pool — add/edit/delete drivers |
| Promotions | CRUD | Discount codes — add/edit/delete |
| Customers | read | Customers scoped to this restaurant |
| Analytics | read | Revenue stats + popular items breakdown |
| Payments | read | Transaction list |

All routes are under `/api/restaurant/*` and require `requireRestaurantAdmin`.

## Partner Onboarding Flow

1. Business submits application at `partner-apply.html`
2. Superadmin approves in `superadmin.html` → system creates a `restaurant_admin` user + Restaurant doc
3. Temp password returned once in API response — superadmin shares it with partner
4. Partner logs in → redirected to `partner-dashboard.html`
